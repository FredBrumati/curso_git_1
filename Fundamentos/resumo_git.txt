✅ Checklist: Iniciando um Projeto com Git

📁 1. Vá até a pasta do projeto
cd nome-da-pasta

🌀 2. Inicie o repositório Git
git init
☑️ Agora a pasta está sendo rastreada pelo Git.

🗂️ 3. Crie qualquer arquivo EX(texto.txt alguma_coisa.py)
→ Necessário para os proximos passos

➕ 4. Adicione os arquivos ao Git
git add .

📝 5. Faça o primeiro commit
git commit -a -m "primeiro commit"

🔀 6. git branch -M main
Esse comando renomeia a branch atual para "main".
git branch -M main

🌐 7. Crie um repositório no GitHub
→ Vá para github.com, clique em "New Repository" e crie um repositório vazio (sem README).

🔗 8. Conecte seu projeto local ao GitHub
git remote add origin (Link fornecido)

🚀 9. Envie seu projeto para o GitHub
git push -u origin master

-------------------------------------------------------------------------------------------------------------------------
🧾 Ordem para envio de um repositório:
git add . → Adiciona todos os arquivos modificados
git commit -a -m "mensagem" → Salva o commit com uma mensagem
git push → Envia o commit para o GitHub
💡 Obs: git commit -a -m não inclui arquivos novos — apenas os que já foram rastreados anteriormente com git add


📌 Comandos principais do Git:
git add (nome do arquivo) → Adiciona um arquivo específico ao projeto
git add . → Adiciona todos os arquivos modificados  📍
git status → Mostra o status atual dos arquivos no projeto
git commit (nome do arquivo) -m "mensagem" → Faz commit de um arquivo com mensagem
git commit -a -m "mensagem" → Faz commit de todos os arquivos já adicionados  📍
git push → Envia os commits locais para o GitHub  📍
git pull → Baixa atualizações do GitHub para o seu projeto local
git clone (link HTTPS) . → Clona um repositório (na pasta atual)
⚠️ Use com cuidado: se a pasta não estiver vazia, pode dar erro. O mais comum é: git clone (link HTTPS)

Manipulação de arquivos:
git rm (nome do arquivo) → Remove um arquivo do repositório
⚠️ Precisa fazer os passos (2) e (3) da ordem de envio após isso.
git mv (arquivo) (pasta/arquivo) → Move um arquivo para uma pasta
⚠️ Também exige commit e push depois.
git mv (arquivo) (novo_nome) → Renomeia um arquivo
⚠️ Também exige commit e push depois.
git checkout (nome do arquivo) → Reverte um arquivo para a última versão salva
✅ Ainda funciona, mas o Git moderno recomenda usar: git restore (nome do arquivo)

🧱 .gitignore
O arquivo .gitignore define quais arquivos e pastas devem ser ignorados pelo Git. 📍
Exemplos:
Ignorar um arquivo:
texto.txt
Ignorar todos os arquivos de uma pasta:
assets/*
⚠️ Todos os arquivos dentro da pasta serão ignorados.

♻️ Reset de repositório
git reset --hard origin/master
→ Restaura o repositório local para o estado do GitHub (apaga commits locais que não foram enviados).

🌿 Branch (Ramificações no Git)
Uma branch é como uma "linha do tempo paralela" do projeto, usada para desenvolver funcionalidades sem interferir no código principal.

🔹 Conceito
Quando um projeto Git é criado, ele inicia na branch padrão chamada master (ou main, no padrão mais atual).
Por padrão, você está trabalhando nela até criar uma nova branch.
Cada nova funcionalidade ou correção geralmente é feita em uma branch separada, para evitar conflitos e manter o código organizado.
🛠️ Vantagens de usar branches:
✅ Permite desenvolver novas ideias sem afetar o código estável
✅ Facilita o trabalho em equipe (cada pessoa pode trabalhar em uma branch diferente)
✅ Ajuda a organizar e revisar as mudanças antes de integrá-las ao projeto principal

-------------------------------------------------------------------------------------------------------------------------
🌿 Branches (Ramificações)
Um branch é como uma linha do tempo separada do projeto. Serve para desenvolver algo sem mexer no código principal.
📌 Comandos principais:
git branch → Lista todos os branches disponíveis.
git branch <nome> → Cria um novo branch.
git branch -d <nome> ou git branch --delete <nome> → Deleta um branch.
⚠️ Não é muito comum deletar branchs locais — normalmente guardamos o histórico.
git checkout <nome> → Muda para outro branch.
git checkout -b <nome> → Cria e já muda para o branch novo.
git merge <nome> → Une o código de outro branch no branch atual.

📦 Git Stash (Guardar alterações temporárias)
O stash é útil quando você está no meio de uma modificação, mas precisa trocar de branch ou pausar o que está fazendo.
📌 Comandos principais:
git stash → Salva as alterações não commitadas em uma “pilha” temporária.
git stash list → Mostra todas as stashes salvas.
git stash show -p <id> → Mostra o que foi modificado na stash.
git stash apply <id> → Recupera uma stash (mantém ela na lista).
git stash drop <id> → Remove uma stash específica.
git stash clear → Remove todas as stashes.
💡 Obs: normalmente o <id> da stash é algo como stash@{0}, stash@{1}, etc.

🏷️ Git Tags (Checkpoints)
As tags funcionam como marcos/checkpoints do projeto. Muito usadas para marcar versões estáveis (ex: v1.0, release-2025).
📌 Comandos principais:
git tag -a <nome> -m "mensagem" → Cria uma nova tag anotada com descrição.
git tag → Lista todas as tags existentes.
git show <nome> → Mostra informações da tag.
git checkout <nome> → Muda para a versão da tag.
git push origin <nome> → Envia uma tag específica para o repositório remoto.
git push origin --tags → Envia todas as tags de uma vez.

🌍 Repositórios Remotos
Um repositório remoto é a versão do seu projeto que está hospedada no GitHub (ou outro servidor).

📌 Comandos principais:
git fetch
→ Atualiza as informações do repositório remoto (branches e tags), mas não altera seus arquivos locais.
💡 Diferente do git pull, ele não faz merge automático.
git remote
→ Lista os repositórios remotos configurados.
git remote -v
→ Mostra os links (URLs) dos repositórios remotos.
git remote add origin <link>
→ Adiciona um repositório remoto.
git remote rm <nome>
→ Remove um repositório remoto configurado.

📦 Submódulos (Submodules)
Submódulos permitem adicionar um repositório Git dentro de outro repositório.
🔹 Muito usado quando você quer reutilizar um projeto externo dentro do seu.

📌 Comandos principais:
git submodule
→ Lista os submódulos existentes no projeto.
git submodule add <link-do-repo>
→ Adiciona um novo submódulo ao projeto.
git push --recurse-submodules=on-demand
→ Envia alterações garantindo que os submódulos também sejam enviados corretamente.
⚠️ Submódulos exigem cuidado extra, pois possuem controle de versão separado do projeto principal.

🔎 Inspeção e Comparação no Git
📖 git show
Exibe informações detalhadas sobre commits, tags e estado atual do repositório.
📌 Uso:
git show
→ Mostra informações do commit atual, incluindo:
Autor
Data
Mensagem
Alterações feitas nos arquivos
git show <tag>
→ Exibe as informações de uma tag específica.
💡 Também pode ser usado com o hash de um commit:
git show <hash-do-commit>

🔍 git diff
Mostra as diferenças entre versões de arquivos ou branches.
📌 Uso:
git diff
→ Mostra alterações feitas nos arquivos ainda não commitados.
git diff <arquivo>
→ Mostra as mudanças feitas em um arquivo específico.
git diff <arquivo1> <arquivo2>
→ Compara dois arquivos diferentes.
💡 Muito usado para revisar mudanças antes de fazer commit.

📊 git shortlog
Exibe um resumo do histórico do projeto.
git shortlog
→ Mostra os commits agrupados por autor.
✔️ Útil para visualizar:
Quantidade de commits por pessoa
Organização da equipe
Resumo geral do histórico

🛠️ Administração e Manutenção do Repositório
🧹 git clean
Remove arquivos não rastreados (untracked), ou seja, arquivos que nunca passaram por git add.
📌 Uso:
git clean -f
→ Remove arquivos não rastreados.
⚠️ O -f (force) é obrigatório para realmente executar a limpeza.
💡 Use com cuidado, pois os arquivos removidos não podem ser recuperados pelo Git.

♻️ git gc (Garbage Collector)
gc significa garbage collector.
git gc
→ Remove arquivos desnecessários e otimiza o repositório.
✔️ Limpa objetos que não estão mais sendo usados
✔️ Melhora performance
✔️ Organiza o armazenamento interno
💡 O Git já executa isso automaticamente em alguns momentos.

🔍 git fsck
Verifica a integridade do repositório.
git fsck
→ Analisa:
Conectividade dos objetos
Possíveis corrupções
Problemas estruturais
✔️ Muito usado para diagnóstico quando algo parece errado no repositório.

📜 git reflog
Registra todas as movimentações feitas no repositório local, incluindo:
Commits
Mudança de branch
Reset
Merge
Rebase
git reflog
→ Mostra um histórico detalhado das ações locais.
💡 Muito útil para recuperar commits perdidos.
⏳ O histórico do reflog normalmente fica disponível por cerca de 30 dias (configurável).

📦 git archive
Permite gerar um arquivo compactado do projeto.
📌 Exemplo:
git archive --format=zip --output=master_files.zip master
→ Gera um arquivo .zip da branch master.
✔️ Muito usado para distribuir o projeto sem incluir o histórico do Git.

📝 A Importância do Commit
🚨 O Problema
Commits sem sentido como:
“alterações” | “teste” | “mudanças” | “ajustes”
❌ Dificultam o entendimento do projeto
❌ Atrapalham revisão de código
❌ Complicam manutenção futura

🎯 Por que padronizar commits?
Padronizar commits ajuda o projeto a crescer de forma saudável.
✔️ Benefícios:
Melhor revisão de Pull Requests
Histórico mais organizado no git log
Facilidade para voltar versões do código
Melhor entendimento do que foi alterado e por quê

🌿 Private Branches (Branches Privadas)
Quando estamos desenvolvendo algo e ainda não queremos compartilhar commits “bagunçados”, podemos:
✔️ Criar uma branch privada
✔️ Trabalhar livremente nela
✔️ Depois organizar os commits antes de enviar

🔄 Rebase Interativo
Após finalizar a funcionalidade, podemos organizar os commits com:
git rebase -i <branch-base>
Exemplo:
git rebase -i main
O modo interativo permite:
🔀 squash → Unir commits
✏️ reword → Alterar mensagem do commit
❌ Remover commits desnecessários
🔄 Reorganizar a ordem dos commits
💡 Isso deixa o histórico mais limpo e profissional antes de compartilhar.

✍️ Boas Mensagens de Commit
Uma boa mensagem de commit deve ser clara e objetiva.
📌 Estrutura recomendada:
🧾 Assunto (título)
Máximo de 50 caracteres
Primeira letra maiúscula
Sem ponto final
📄 Corpo (opcional, mas recomendado)
Máximo de 72 caracteres por linha
Explicar o porquê da mudança
Explicar o que foi feito
❗ Não explicar linha por linha do código

🔗 Referência de Padrão de Commits
Para padronização profissional de mensagens de commit, consulte:
📘 Conventional Commits (documentação oficial):
https://www.conventionalcommits.org/pt-br/v1.0.0-beta.4/

🌐 GitHub — Estrutura do Repositório

📦 Criando um Repositório
No GitHub criamos repositórios preenchendo algumas informações importantes:
📌 Informações iniciais:
Nome do repositório
Descrição
Licença
💡 Essas informações podem ser alteradas depois, mas é importante configurá-las corretamente no início do projeto.
📑 Principais Abas do GitHub

💻 Aba Code
Na aba Code encontramos as principais informações do repositório:
Código fonte do projeto
Arquivo README.md (documentação)
Arquivo de licença
Criação de branches
Upload de arquivos
📌 É a aba mais utilizada no dia a dia.

🐞 Aba Issues
A aba Issues serve para organizar tarefas e problemas do projeto.
Podemos usar para:
Reportar bugs
Criar tarefas
Planejar novas funcionalidades
💡 Boas práticas:
Cada issue deve ter uma label
Deve ter um responsável
Podemos usar Markdown na descrição

🔀 Aba Pull Requests
A aba Pull Requests é usada para enviar alterações de código para o projeto.
📌 Fluxo comum:
1️⃣ Criar um branch
2️⃣ Fazer alterações no código
3️⃣ Enviar para o repositório
4️⃣ Criar um Pull Request
⚠️ O código não deve ir direto para a branch principal, ele deve ser revisado antes.

⚙️ Aba Actions
A aba Actions permite criar automatizações no projeto.
Exemplos:
Integração contínua (CI)
Deploy automático (CD)
Execução de testes automáticos
💡 Muito utilizada para automatizar processos do projeto.

📋 Aba Projects
Na aba Projects podemos criar quadros de organização de tarefas.
Esse modelo utiliza o sistema Kanban.
📌 Estrutura comum:
Backlog
Em desenvolvimento
Em teste
Finalizado
💡 A interface é parecida com o Trello.

📚 Aba Wiki
A aba Wiki permite criar uma documentação mais detalhada do projeto.
Podemos documentar:
Funcionalidades
Bugs conhecidos
Informações técnicas
Guias de uso
📌 Funciona como uma base de conhecimento do projeto.

📊 Aba Insights
A aba Insights mostra estatísticas do projeto.
Exemplos:
Contribuidores
Commits
Forks
Atividade do repositório
💡 Útil para acompanhar a evolução do projeto.

⚙️ Aba Settings
Na aba Settings encontramos as configurações do repositório.
Podemos:
Alterar o nome do repositório
Adicionar colaboradores
Ativar ou desativar funcionalidades
Excluir o repositório
⚠️ A exclusão do repositório também é feita nesta aba.

🧩 GitHub Gist
Gist são pequenos blocos de código hospedados no GitHub.
Podemos usar para:
Guardar soluções rápidas
Compartilhar trechos de código
Salvar exemplos úteis
📌 Cada Gist funciona como um pequeno repositório com um link compartilhável.
